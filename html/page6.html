The work process

Software development process

Historically, software development followed the waterfall process. This is a sequential process. First the software requirements specification (SRS) document was established. Then the software architecture was decided and described in an architecture describtion language (ADL), the unified modeling language (UML), or the system modeling language (SysML). Implementation follows, then testing and integration, deployment and maintenance.

In practice, such a sequential workflow is almost impossible to follow. The V-model is an extension of the waterfall model. The left-to-right axis represents time and the vertical axis level of abstraction. The conventional V-model has horizontal arrows directed in the time direction, but in the V-model figure below, I have used two-way horizontal arrows. This is in recognition of the iterative nature of software development. More often than not, issues found on the right-hand-side of the V forces one to rethink stuff on the left-hand-side.

V-model

The last decades agile development processes such as scrum have become popular. They adopt a people-centric perspective and incorporate iteration and continuous feedback. The spiral model illustrated in the figure below. Prototypes always evolve over several revolutions of the spiral. In the figure only one prototype is developed. This development model is advisable if the users are unsure of their needs, the requirements are complex, or if significant a amount of research and exploration is needed.

Spiral-model

Whatever development model is used, the most important thing is to have a clear specification of the required functionality when implementation starts. Without knowing the target, it will be hard to catch. This is not to say that the target will be fixed at the outset. Often in research oriented projects, the target moves during development, but to avoid delays and extra costs, the target should be more or less fixed at the end of the prototyping stage.

Version control

Turning from the process on the macroscopic project level, let us look at the day-to-day process. Nowadays, almost every software project from the smallest hobby project to large corporate projects use a software version control system.

Consider the version control workflow illustrated below. There is a main developer branch and a release branch. Developer A clones from the main branch baseline \(M_0\) and gets version \(A_0\). When A is finished with her work, she submits her work \(A_1\) for pushing into the main branch.

Version control workflow

The developer submits the candidate to review. This submission triggers a selected set of builds and tests by an automation server. It is important that this test suite can be built and run in a few minutes. If the tests fail, the submission is not approved and the developer has to fix the bugs and submit once more. When the pre-review automation server tests pass, the code is sent to a review board for evaluation. As soon as the review board approves the submission, the automation server should trigger builds on all relevant platforms as well as running all available unit tests, integration tests, and system tests. When all tests pass, a new sound baseline is created. This process guarantees that every baseline of the master builds and passes all tests.

Pushing to main

If the full test process takes too much time, one may have to adopt a faster integration scheme, where full test suits are carried out after baseline creation instead of before. In this case, there is no guarantee that a baseline passes all tests. If this process is adopted, it may be useful to have a separate branch into which only sound baselines are merged automatically.

Use a separate release branch for each software release. When the release branch is ready for deployment, it should receive no more pushes from the main development branch. Any bugfixes should be done on bugfix activities (e.g., \(\text{Bug}_0\) to \(\text{Bug}_1\). taken from the release branch and pushed back to it and to the main branch. At the same time new development can continue on the main branch. The choice of work process is not critical. However, the process should ensure that all code pass a peer review process. It should also guarantee that all tests are run for all pushes to the main branch.
